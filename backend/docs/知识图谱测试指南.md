# 知识图谱功能测试指南

## 功能概述

本项目已成功集成知识图谱功能，使用 LangChain 的 `LLMGraphTransformer` 实现自动化的知识图谱构建。

## 核心特性

### 1. Schema 约束
- **法律文档 Schema**: 定义了 10 种节点类型和 14 种关系类型
- **案例文档 Schema**: 定义了 9 种节点类型和 10 种关系类型
- **严格模式**: 确保只提取预定义的实体和关系类型

### 2. 自动化构建
- 上传文档自动提取实体和关系
- 支持多种文档格式（PDF、Word、文本）
- 自动分块处理长文档

### 3. 图谱查询
- 实体查询：基于实体名称查询相关节点
- 关系查询：基于关系类型查询相关关系
- Cypher 查询：支持原生 Neo4j Cypher 查询

## API 接口

### 1. 上传文档构建图谱

**接口**: `POST /api/v1/graph/upload`

**请求参数**:
- `file`: 上传的文件（PDF、Word、文本）
- `document_type`: 文档类型（legal/case/general）
- `strict_mode`: 是否启用严格模式（默认 true）

**示例**:
```bash
curl -X POST "http://localhost:8000/api/v1/graph/upload" \
  -F "file=@劳动合同法.pdf" \
  -F "document_type=legal" \
  -F "strict_mode=true"
```

**响应示例**:
```json
{
  "success": true,
  "message": "知识图谱构建成功",
  "code": 200,
  "file_name": "劳动合同法.pdf",
  "document_type": "legal",
  "nodes_count": 15,
  "relationships_count": 8
}
```

### 2. 查询知识图谱

**接口**: `POST /api/v1/graph/query`

**请求参数**:
- `query_type`: 查询类型（entity/relation/cypher）
- `query`: 查询内容
- `entity_type`: 实体类型（可选）
- `relation_type`: 关系类型（可选）
- `limit`: 返回结果数量限制（默认 10）

**示例 1: 实体查询**
```bash
curl -X POST "http://localhost:8000/api/v1/graph/query" \
  -H "Content-Type: application/json" \
  -d '{
    "query_type": "entity",
    "query": "劳动合同法第39条",
    "entity_type": "LawArticle",
    "limit": 10
  }'
```

**示例 2: 关系查询**
```bash
curl -X POST "http://localhost:8000/api/v1/graph/query" \
  -H "Content-Type: application/json" \
  -d '{
    "query_type": "relation",
    "relation_type": "APPLIES_LAW",
    "entity_type": "LegalCase",
    "limit": 10
  }'
```

**示例 3: Cypher 查询**
```bash
curl -X POST "http://localhost:8000/api/v1/graph/query" \
  -H "Content-Type: application/json" \
  -d '{
    "query_type": "cypher",
    "query": "MATCH (c:LegalCase)-[r:APPLIES_LAW]->(l:LawArticle) RETURN c, l LIMIT 10"
  }'
```

### 3. 获取统计信息

**接口**: `GET /api/v1/graph/statistics`

**示例**:
```bash
curl -X GET "http://localhost:8000/api/v1/graph/statistics"
```

**响应示例**:
```json
{
  "success": true,
  "message": "获取统计信息成功",
  "code": 200,
  "total_nodes": 150,
  "total_relationships": 80,
  "node_types": [
    {"type": "LawArticle", "count": 50},
    {"type": "LegalCase", "count": 30},
    {"type": "Party", "count": 40}
  ],
  "relationship_types": [
    {"type": "APPLIES_LAW", "count": 30},
    {"type": "INVOLVES", "count": 25}
  ]
}
```

### 4. 清空图谱

**接口**: `DELETE /api/v1/graph/clear`

**示例**:
```bash
curl -X DELETE "http://localhost:8000/api/v1/graph/clear"
```

## 支持的节点类型

### 法律文档 Schema

| 节点类型 | 说明 | 主要属性 |
|---------|------|---------|
| LawArticle | 法条 | name, code, content, category |
| LegalConcept | 法律概念 | name, description, category |
| LegalTerm | 法律术语 | name, definition, usage |
| LegalCase | 案例 | name, case_no, summary, outcome |
| Party | 当事人 | name, role, type |
| Court | 法院 | name, level, jurisdiction |
| LegalDocument | 法律文书 | name, type, content |
| LegalProcedure | 法律程序 | name, description, duration |
| LegalSubject | 法律主体 | name, type, capacity |
| LegalObject | 法律客体 | name, type, value |

### 案例文档 Schema

| 节点类型 | 说明 | 主要属性 |
|---------|------|---------|
| LegalCase | 案例 | name, case_no, summary, outcome |
| Party | 当事人 | name, role, type |
| LawArticle | 法条 | name, code, content |
| Court | 法院 | name, level, location |
| Evidence | 证据 | name, type, description |
| LegalBasis | 法律依据 | name, content, relevance |
| Judgment | 判决 | content, result, amount |
| Lawyer | 律师 | name, license_no, firm |
| Agent | 代理人 | name, type, authorization |

## 注意事项

1. **大模型成本**: 每次上传都会调用大模型进行实体和关系提取，请注意成本控制
2. **文档长度**: 当前限制为处理完整文档，长文档会自动分块
3. **数据质量**: 大模型识别结果可能需要人工审核
4. **性能优化**: 可以考虑缓存常用查询结果

## 后续优化方向
