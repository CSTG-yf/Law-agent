## 基础向量检索 (Basic)

### 原理

基于向量相似度的语义检索，将查询文本转换为向量，在向量数据库中查找最相似的文档。

```
查询文本 → 向量化 → 余弦相似度计算 → 返回 Top-K 结果
```

### 特点

| 优点 | 缺点 |
|------|------|
| 理解语义关系 | 无法精确匹配关键词 |
| 跨语言检索 | 计算成本较高 |
| 适合概念性查询 | 对专业术语敏感度低 |

### 适用场景

- 概念性查询："什么是劳动仲裁？"
- 语义相似查询："员工离职补偿"
- 跨语言检索

## 混合检索 (Hybrid)

### 原理

结合向量检索和 BM25 关键词检索，使用倒数排名融合（RRF）合并结果。

```
查询 → 向量检索 ─┐
              ├→ RRF 融合 → 排序 → 返回结果
查询 → BM25 检索 ─┘
```

### RRF 融合算法

倒数排名融合（Reciprocal Rank Fusion）的核心思想是：

```
score(doc) = Σ (1 / (rank_i + k)) × weight_i
```

- `rank_i`: 文档在第 i 个检索方法中的排名
- `k`: 平滑常数（通常为 60）
- `weight_i`: 第 i 个检索方法的权重

**示例**：

| 文档 | 向量检索排名 | BM25 排名 | 向量权重 0.7 | BM25 权重 0.3 | 总分 |
|------|-------------|-----------|---------------|---------------|------|
| 文档A | 1 | 3 | 1/61×0.7=0.0115 | 1/63×0.3=0.0048 | 0.0163 |
| 文档B | 2 | 1 | 1/62×0.7=0.0113 | 1/61×0.3=0.0049 | 0.0162 |
| 文档C | 3 | 2 | 1/63×0.7=0.0111 | 1/62×0.3=0.0048 | 0.0159 |

按总分排序：文档A > 文档B > 文档C

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `vector_weight` | 0.7 | 向量检索权重 |
| `bm25_weight` | 0.3 | BM25 检索权重 |

### 特点

| 优点 | 缺点 |
|------|------|
| 兼顾语义和关键词 | 计算复杂度较高 |
| 提高召回率 | 需要调优权重 |
| 适合专业领域 | 需要维护 BM25 索引 |

### 适用场景

- 法律文档检索（专业术语多）
- 技术文档查询
- 需要精确匹配关键词的场景

## MMR 多样性检索 (MMR)

### 原理

最大边际相关性（Maximal Marginal Relevance），平衡相关性和多样性。

```
初始检索 → 迭代选择 → 计算相关性 → 计算多样性 → MMR 评分 → 选择最佳
```

### MMR 评分公式

```
MMR(D) = λ × Sim(D, Q) - (1 - λ) × max(Sim(D, Di))
```

| 参数 | 说明 |
|------|------|
| `D` | 候选文档 |
| `Q` | 查询 |
| `Di` | 已选中的文档 |
| `λ` | 多样性因子 (0-1) |
| `Sim()` | 相似度函数 |

### 迭代选择过程

MMR 通过贪心算法迭代选择文档：

```
步骤1: 初始检索 fetch_k 个文档
步骤2: 计算每个候选文档的 MMR 分数
步骤3: 选择 MMR 分数最高的文档
步骤4: 从候选集中移除该文档
步骤5: 重复步骤2-4，直到选出 k 个文档
```

**示例**：

假设查询"劳动合同"，初始检索到 5 个文档，λ=0.5：

| 轮次 | 候选文档 | 相关性 | 多样性 | MMR 分数 | 选择 |
|------|----------|--------|--------|----------|------|
| 1 | 文档A | 0.9 | 1.0 | 0.5×0.9+0.5×1.0=0.95 | ✓ |
| 1 | 文档B | 0.85 | 1.0 | 0.5×0.85+0.5×1.0=0.925 | - |
| 2 | 文档B | 0.85 | 0.3 | 0.5×0.85+0.5×0.3=0.575 | ✓ |
| 2 | 文档C | 0.8 | 0.7 | 0.5×0.8+0.5×0.7=0.75 | - |

最终选择：文档A、文档B

### 参数说明

| 参数 | 范围 | 效果 |
|------|------|------|
| `lambda_mult` | 0-1 | 1=纯相关性，0=纯多样性 |
| `fetch_k` | >k | 初始获取数量（建议 20-50） |

### λ 值对比

| λ 值 | 效果 | 适用场景 |
|------|------|----------|
| 1.0 | 纯相关性 | 精确查询 |
| 0.7 | 偏向相关性 | 大部分场景 |
| 0.5 | 平衡 | 推荐 |
| 0.3 | 偏向多样性 | 探索性查询 |
| 0.0 | 纯多样性 | 发现性查询 |

### 特点

| 优点 | 缺点 |
|------|------|
| 避免结果重复 | 计算成本高 |
| 覆盖不同角度 | 需要调优参数 |
| 适合探索性查询 | 可能降低相关性 |

### 适用场景

- 法律条款检索（避免重复条款）
- 多角度问题分析
- 需要全面了解主题

## 多查询检索 (Multi-Query)

### 原理

生成多个查询变体，分别检索后合并结果，提高召回率。

```
原始查询 → 生成变体 → 多路检索 → 结果合并 → 去重排序 → 返回
```

### 查询变体生成策略

多查询检索通过以下方法生成查询变体：

**方法1：关键词提取**
- 使用分词工具（如 jieba）提取关键词
- 将关键词重新组合形成新查询
- 示例："劳动合同解除赔偿" → "劳动合同 解除 赔偿"

**方法2：同义词扩展**
- 基于同义词词典扩展查询
- 替换查询中的关键词为同义词
- 示例："赔偿" → "补偿"、"赔偿金"

**方法3：重写查询**
- 使用 LLM 重写查询
- 生成语义等价但表达不同的查询
- 示例："如何申请劳动仲裁" → "劳动仲裁申请流程"

### 结果合并策略

多个查询检索到的结果需要合并和去重：

**累加分数法**：
```
score(doc) = Σ (1 / (distance + ε)) × weight
```

- `distance`: 文档与查询的距离
- `ε`: 平滑常数（避免除零）
- `weight`: 查询权重

**投票法**：
- 统计每个文档被多少个查询命中
- 命中次数越多，排名越靠前

**示例**：

| 文档 | 查询1命中 | 查询2命中 | 查询3命中 | 累加分数 | 最终排名 |
|------|----------|----------|----------|----------|----------|
| 文档A | ✓ | ✓ | - | 0.95+0.88=1.83 | 1 |
| 文档B | ✓ | - | ✓ | 0.92+0.85=1.77 | 2 |
| 文档C | - | ✓ | ✓ | 0.90+0.82=1.72 | 3 |

### 查询变体示例

| 原始查询 | 变体1 | 变体2 | 变体3 |
|----------|--------|--------|--------|
| "劳动合同解除赔偿" | "劳动合同 解除 赔偿" | "经济补偿金" | "离职补偿标准" |
| "劳动仲裁流程" | "劳动仲裁 流程" | "仲裁程序" | "仲裁申请" |

### 特点

| 优点 | 缺点 |
|------|------|
| 提高召回率 | 查询时间增加 |
| 覆盖多种表达 | 结果可能不相关 |
| 适合探索性查询 | 需要设计变体策略 |

### 适用场景

- 用户查询表达不明确
- 需要全面了解主题
- 探索性查询

## 重排序 (Reranking)

### 原理

在初步检索结果基础上，使用更精确的模型重新排序，提高结果质量。

```
初始检索 → 候选文档 → 重排序模型 → 精确评分 → 返回 Top-K
```

### 为什么需要重排序

#### 问题：向量检索的局限性

向量检索使用 **Bi-Encoder**（双向编码器）：

```
查询: "劳动合同解除赔偿"
  ↓ [向量化]
查询向量: [0.1, 0.2, 0.3, ...]

文档A: "劳动合同解除需要支付经济补偿金"
  ↓ [向量化]
文档A向量: [0.12, 0.18, 0.28, ...]

相似度 = cosine(查询向量, 文档A向量) = 0.85
```

**局限性**：
- 查询和文档独立编码，无法捕获交互信息
- 向量空间中的相似度不等于语义相关性
- 对长文档、复杂查询效果差

#### 解决方案：Cross-Encoder 重排序

使用 **Cross-Encoder**（交叉编码器）：

```
输入: [CLS] 查询 [SEP] 文档 [SEP]
  ↓ [联合编码]
输出: 相关性分数 = 0.92
```

**优势**：
- 查询和文档联合编码，捕获交互信息
- 直接计算相关性分数，更精确
- 适合 Top-K 场景

### Bi-Encoder vs Cross-Encoder 对比

| 特性 | Bi-Encoder | Cross-Encoder |
|------|-------------|---------------|
| 编码方式 | 独立编码 | 联合编码 |
| 计算复杂度 | O(1) 预计算 + O(N) 检索 | O(N) 实时计算 |
| 检索速度 | 快 | 慢 |
| 相关性精度 | 中 | 高 |
| 适用场景 | 大规模检索 | 精细化排序 |
| 典型模型 | BERT, RoBERTa | Cross-Encoder |

### 两阶段检索流程

```
阶段1: 快速检索 (Bi-Encoder)
  查询 → 向量化 → 向量数据库 → 返回 100 个候选文档

阶段2: 精确重排序 (Cross-Encoder)
  100 个候选文档 → Cross-Encoder → 评分 → 返回 Top-10
```

**优势**：
- 兼顾速度和精度
- 适合大规模文档库
- 可扩展性强

### Cross-Encoder 架构

```
输入层
  ↓
[CLS] 查询 [SEP] 文档 [SEP]
  ↓
嵌入层 (Embedding Layer)
  ↓
Transformer 编码器 (多层注意力机制)
  ↓
池化层 (Pooling)
  ↓
全连接层 (FC)
  ↓
输出层 (Sigmoid)
  ↓
相关性分数 (0-1)
```

### 数学原理

#### 注意力机制

Cross-Encoder 的核心是自注意力机制：

```
Attention(Q, K, V) = softmax(QK^T / √d_k) V
```

- **Q (Query)**: 查询的表示
- **K (Key)**: 文档的表示
- **V (Value)**: 文档的内容
- **d_k**: 向量维度

通过注意力机制，模型能够：
- 捕获查询和文档之间的交互
- 识别关键词匹配
- 理解上下文关系

#### 训练目标

Cross-Encoder 通常使用对比学习训练：

```
损失函数 = -log(exp(sim(q, d_pos)) / (exp(sim(q, d_pos)) + Σexp(sim(q, d_neg)))
```

- `q`: 查询
- `d_pos`: 正样本（相关文档）
- `d_neg`: 负样本（不相关文档）
- `sim()`: 相似度函数

### 重排序流程

重排序的完整流程如下：

```
输入：查询 + 候选文档列表
  ↓
构建查询-文档对：(query, doc1), (query, doc2), ...
  ↓
Cross-Encoder 编码：对每对进行联合编码
  ↓
输出相关性分数：[0.95, 0.88, 0.82, ...]
  ↓
按分数排序：doc1 > doc2 > doc3 > ...
  ↓
返回 Top-K 结果
```

**关键点**：
- 候选文档数量通常为 50-100 个
- Cross-Encoder 对每对独立计算
- 最终返回 Top-K（如 10 个）

### 模型选择

| 模型 | 大小 | 速度 | 精度 | 适用场景 |
|------|------|------|------|----------|
| `ms-marco-MiniLM-L-6-v2` | 小 | 快 | 中 | 实时应用 |
| `ms-marco-MiniLM-L-12-v2` | 中 | 中 | 高 | 通用场景 |
| `ms-marco-electra-base` | 中 | 中 | 高 | 英文场景 |
| `bge-reranker-base` | 中 | 中 | 高 | 中文场景 |

### 特点

| 优点 | 缺点 |
|------|------|
| 提高结果质量 | 增加计算成本 |
| 更精确的相关性 | 需要加载额外模型 |
| 适合 Top-K 场景 | 延迟增加 |

### 适用场景

- 对结果质量要求高
- Top-K 推荐
- 搜索引擎

## 策略对比与选择

### 性能对比

| 策略 | 召回率 | 精确度 | 速度 | 多样性 | 复杂度 |
|------|--------|--------|------|--------|--------|
| Basic | 中 | 中 | ⭐⭐⭐⭐⭐ | 低 | 低 |
| Hybrid | 高 | 高 | ⭐⭐⭐⭐ | 中 | 中 |
| MMR | 中 | 中 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 高 |
| Multi-Query | ⭐⭐⭐⭐⭐ | 中 | ⭐⭐ | 中 | 中 |
| +Rerank | - | ⭐⭐⭐⭐⭐ | ⭐⭐ | - | 高 |

### 选择建议

| 场景 | 推荐策略 |
|------|----------|
| 快速响应 | Basic |
| 法律文档检索 | Hybrid + Rerank |
| 多角度分析 | MMR |
| 探索性查询 | Multi-Query |
| 高质量结果 | Hybrid + Rerank |

### 参数调优建议

#### Hybrid 检索

```python
# 法律文档：关键词重要
vector_weight = 0.6
bm25_weight = 0.4

# 通用场景：语义重要
vector_weight = 0.8
bm25_weight = 0.2
```

#### MMR 检索

```python
# 精确查询
lambda_mult = 0.7
fetch_k = 20

# 平衡查询（推荐）
lambda_mult = 0.5
fetch_k = 30

# 探索性查询
lambda_mult = 0.3
fetch_k = 50
```

#### Multi-Query 检索

```python
# 生成 3-5 个查询变体
max_queries = 5

# 每个查询获取更多结果
results_per_query = k * 2
```

## API 使用示例

### Python SDK

```python
from app.service.rag.hybrid_retriever import AdvancedRAGService
from app.service.rag.reranker import Reranker
from app.service.vector_db import ChromaVectorStore

vector_store = ChromaVectorStore(collection_name="legal_documents")
rag_service = AdvancedRAGService(vector_store)
reranker = Reranker()

# 混合检索 + 重排序
docs = rag_service.search(
    query="劳动合同解除赔偿",
    strategy="hybrid",
    k=5
)

docs = reranker.rerank(
    query="劳动合同解除赔偿",
    documents=docs,
    top_k=5
)
```

### REST API

```bash
# 基础检索
curl -X POST http://localhost:8000/api/v1/rag/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "劳动合同解除赔偿",
    "n_results": 5,
    "strategy": "basic"
  }'

# 混合检索 + 重排序
curl -X POST http://localhost:8000/api/v1/rag/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "劳动合同解除赔偿",
    "n_results": 5,
    "strategy": "hybrid",
    "enable_rerank": true
  }'

# MMR 检索
curl -X POST http://localhost:8000/api/v1/rag/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "劳动合同解除赔偿",
    "n_results": 5,
    "strategy": "mmr",
    "fetch_k": 30,
    "lambda_mult": 0.5
  }'
```

## 最佳实践

1. **从简单开始**：先用 Basic 策略，根据需求逐步升级
2. **监控性能**：记录各策略的召回率、精确度、响应时间
3. **A/B 测试**：对比不同策略的效果
4. **参数调优**：根据业务场景调整权重和参数
5. **组合使用**：Hybrid + Rerank 通常效果最佳
